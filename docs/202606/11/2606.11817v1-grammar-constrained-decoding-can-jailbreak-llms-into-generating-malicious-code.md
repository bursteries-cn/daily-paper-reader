## 摘要
大型语言模型（LLMs）越来越多地被用于代码生成，引发了它们可能被滥用生成恶意代码的担忧。与此同时，语法约束解码（GCD）已被广泛采用，通过强制语法有效性来提高LLM生成代码的可靠性。在本文中，我们揭示了一个反直觉的风险：这种面向可靠性的技术本身可能成为一个攻击面。我们发现了一种新的越狱攻击，称为CodeSpear，它利用GCD诱导LLM生成恶意代码。我们的实验表明，仅仅应用一个良性的代码语法约束就能有效地越狱LLM。为了解决这一漏洞，我们提出了CodeShield，一种安全对齐方法，即使在攻击者控制的语法约束下也能稳健地保持安全行为。CodeShield通过教模型在GCD下生成蜜罐代码来对齐代码模态。这种代码在语义上是无害的，因此不会执行恶意请求，并且在结构上多样化，因此难以通过收紧语法来抑制。同时，当自然语言可用时，CodeShield仍然保留自然语言的拒绝。在4个基准测试中对10个流行的LLM进行的实验表明，CodeSpear优于代表性的越狱基线，平均攻击成功率提高了30个百分点以上。CodeShield在保持良性效用的同时，也恢复了在CodeSpear下的安全性。我们的发现揭示了GCD的基本风险，并呼吁对其潜在的安全影响给予更多关注。

## Abstract
Large Language Models (LLMs) are increasingly used for code generation, raising concerns that they may be misused to produce malicious code. Meanwhile, Grammar-Constrained Decoding (GCD) has been widely adopted to improve the reliability of LLM-generated code by enforcing syntactic validity. In this paper, we reveal a counterintuitive risk: this reliability-oriented technique can itself become an attack surface. We uncover a new jailbreak attack, termed CodeSpear, that exploits GCD to induce LLMs into generating malicious code. Our experiments show that simply applying a benign code grammar constraint can effectively jailbreak LLMs.   To address this vulnerability, we propose CodeShield, a safety alignment approach that robustly preserves safe behavior even under attacker-controlled grammar constraints. CodeShield aligns the model in the code modality by teaching it to generate honeypot code under GCD. Such code is semantically harmless, so it does not implement the malicious request, and structurally diverse, so it is difficult to suppress through grammar tightening. At the same time, CodeShield still preserves natural-language refusals when natural language is available. Experiments on 10 popular LLMs across 4 benchmarks show that CodeSpear outperforms representative jailbreak baselines and increases the attack success rate by more than 30 percentage points on average. CodeShield also restores safety under CodeSpear while preserving benign utility. Our findings reveal a fundamental risk of GCD and call for greater attention to its potential security implications.

## 1. 论文的核心问题与整体含义（研究动机和背景）

- **问题**：语法约束解码（Grammar-Constrained Decoding, GCD）原本是为了提升 LLM 生成代码的可靠性，但本文发现它意外地成为新的攻击面——攻击者只需使用一个**标准、良性**的代码语法约束，就能**诱导 LLM 生成恶意代码**，而无需精心构造对抗性语法、梯度优化或提示工程。
- **背景**：现有安全对齐几乎完全建立在自然语言模态上，教会模型输出自然语言的拒绝回答（如“我不能帮助”）。然而，GCD 将有效输出空间限制为代码语法定义的序列，自然语言拒绝通常落在代码语法之外，因此模型被迫在代码模态中继续生成——而代码模态没有经过显式的安全对齐，导致模型“被迫”生成有害内容。
- **整体含义**：揭示了 GCD 这一可靠性增强机制的根本性安全风险，需要重新审视 LLM 安全对齐对输出模态的假设。

### 2.1 CodeSpear（攻击方法）

- **关键技术细节**：
  - 给定一个恶意提示 \( p \) 和标准代码语法 \( G \)（如 Python 语法）。
  - 使用 GCD 接口，每一步只允许生成能扩展为有效代码的 token（公式 2-3）。
  - 输出必然属于 \( L(G) \)，自然语言拒绝被排除在外（公式 7）。
  - **不需要**对抗语法、模型微调或梯度优化；仅调用现有 GCD 接口。
- **算法流程**（Algorithm 1）：循环解码，每步计算有效 token 集，采样下一个 token，直到遇到序列结束标记。

### 2.2 CodeShield（防御方法）
- **核心思想**：在代码模态中也进行安全对齐，教模型在 GCD 下输出**蜜罐代码（honeypot code）**——语义无害、结构多样的代码，既不会实现恶意要求，又难以被语法收紧移除。
- **关键技术细节**：
  - 使用 DPO（Direct Preference Optimization）进行对齐。
  - 构建三类响应：自然语言拒绝 \( y_{\text{refuse}} \)、蜜罐代码 \( y_{\text{honeypot}} \)、有害代码 \( y_{\text{harmful}} \)。
  - 偏好层次：\( y_{\text{refuse}} \succ y_{\text{honeypot}} \)（自然语言可用时）、\( y_{\text{honeypot}} \succ y_{\text{harmful}} \)（仅代码空间时）。
  - 偏好对构建方式：
    - \( y_{\text{harmful}} \)：用 GCD 查询目标模型获得。
    - \( y_{\text{honeypot}} \)：从通用代码语料库（如 OpenCodeInstruct）中随机采样 K 个不相关的代码片段。
    - 为每个提示生成 K 对 \( (p, y_{\text{refuse}}, y_{\text{honeypot}}) \) 和 \( (p, y_{\text{honeypot}}, y_{\text{harmful}}) \)。
- **训练**：使用标准 DPO 损失（公式 6），同时加入了 40k 通用代码生成任务进行 SFT 以保持效用。
- **直接输出**：训练后，模型在自然语言可用时仍输出拒绝；在 GCD 下则倾向于输出蜜罐代码。

## 6. 论文的主要结论与发现

1. **CodeSpear 攻击非常有效**：
   - 在本地部署模型上，平均 ASR 从 54.92% 提升到 81.82%（+26.90 百分点），优于所有基线。
   - 在 API 模型上，平均 ASR 从 22.00% 提升到 67.39%（+45.39 百分点），同样远超其他方法。
2. **固定安全模式脆弱**：GPT-5 系列在 GCD 下倾向于输出 `pass` 语句，但被语法收紧轻易绕过（ASR 从 55% 升到 70%+）。
3. **CodeShield 能有效防御**：
   - 在 CodeSpear 攻击下，ASR 从 80%+ 降至 5%~8%。
   - 保留了自然语言拒绝能力；在没有 CodeSpear 时进一步降低了原始 ASR。
   - 标准 Safe-DPO（仅自然语言拒绝）在 GCD 下几乎无效，证明蜜罐代码对齐的必要性。
4. **CodeShield 良性效用保持较好**：pass@k 仅有轻微下降（部分场景甚至提升），安全税可接受。
5. **自适应攻击下仍然稳健**：自适应语法收紧后 ASR 未显著上升，证明蜜罐代码的结构多样性使攻击者难以彻底排除。

## 7. 优点

- **攻击方法新颖且现实**：首次系统揭示 GCD 本身可作为攻击面，攻击成本极低（仅需标准 GCD 接口），威胁真实（VLLM、SGLang、OpenAI 等均支持 GCD）。
- **防御设计精巧**：蜜罐代码同时满足“语义无害”和“结构多样性”两个关键要求，且与 DPO 天然结合；保留了双模态安全行为。
- **实验全面**：覆盖本地和 API 两种部署、多种模型系列、多种基线、多维度消融及自适应攻击，结论具有较强的说服力。
- **分析到位**：通过语法收紧实验明确指出固定模式（如 `pass`）的不安全性，强化了防御设计动机。
- **开源贡献**：承诺公开 CodeShield 代码，只向研究者提供 CodeSpear 代码以控制风险。
